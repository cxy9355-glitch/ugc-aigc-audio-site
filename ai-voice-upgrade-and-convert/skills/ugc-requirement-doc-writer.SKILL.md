# UGC需求文档写作器

用于按用户现有的 UGC 技术策划风格，先完成文档框架与流程判断，再在条件满足时推动交互原型，最后撰写 Markdown 版需求文档。

## 职责边界

- 这个 skill 不是一上来就直接写正文的工具。
- 这个 skill 的职责是先落文档框架、判断需求是否涉及界面或场景改动、判断是否已经具备开始画交互原型的条件。
- 如果判断需要交互原型，应该先推动进入交互原型阶段，而不是继续发散写正文。
- 只有当交互原型已经足够支撑正式文档时，才进入正文撰写阶段。
- 对正式策划文档来说，`USLOG/数据埋点` 章节通常放在文档最后一章统一补充，作为收尾步骤处理，不需要在前面搭框架时过早展开细节。

## 工作流程

1. 先只阅读最少量的既有需求文档，用来判断用户的语气、编号方式和结构习惯。
2. 在写任何正文前，必须先把整篇文档框架搭好。要先确定章节结构、各章边界，以及每一章负责解决什么问题，不能直接跳进零散正文。
3. 在搭框架时，同时判断本次需求是否涉及 UI 界面修改、游戏内场景改动，或其它明显需要交互原型支撑的内容。
4. 如果判断需要交互原型，就先确认本次涉及哪些界面的新增、修改、删除，哪些场景表现会被改动，并据此整理原型范围。
5. 如果当前信息不足以支撑原型范围判断，就必须停下来向策划索要 Figma、截图、录屏、视频、草图或场景说明，不能继续发散写正文。
6. 一旦确认信息足够开始画原型，就应先进入交互原型阶段，而不是继续补文档正文。
7. 只有在交互原型已经足够支撑正式文档时，才继续按既有风格补正文：标题简短、规则直给、少讲理论、难点规则下方紧跟示例。
8. 优先使用“规则先行”的写法，直接写清楚什么情况下发生什么、用户会看到什么。
9. 默认输出 Markdown，除非用户明确要求其它格式。
10. 在正式文档接近完成时，最后统一检查并补充 `USLOG/数据埋点` 章节，作为文档收尾步骤。

## 行文风格

- 开头固定从 `1.1 需求背景` 和 `1.2 功能概述` 起。
- 在 `1.2 功能概述` 之后，大章节重新从 2 开始编号，例如 `2.1`、`2.2`、`2.3`。
- 使用层级编号，例如 `1.1`、`2.1`、`2.1.1`，不要使用松散标题体系。
- “先搭框架，再写正文”是硬规则，不是建议。必须先产出章节骨架，再逐章填规则。
- 如果需求涉及 UI 或游戏场景改动，交互章节也属于框架的一部分，不能等正文快写完了再补。
- 语言风格保持精炼中文 PRD 写法，优先写结论和规则，不写大段解释性废话。
- 每条规则都尽量短、准、可执行。普通规则列表优先用 `1）2）3）`。
- 对判断链，不要为了写 if-else 硬拆成 `2.x.5.1 / 2.x.5.2 / 2.x.5.3` 这种小节。应在同一小节下用无序子弹直接展开。
- 除非本次功能真的改变了背景、归属或治理信息，否则不要重复写这些已知信息。
- 除非用户明确要求，不要加“其他说明”这种兜底大节。
- 如果一条规则比较抽象或容易误解，要在该规则正下方直接补 `例如：`，不要把示例统一堆到后面。

## 文档命名规则

- 正式文档文件名统一使用：`U5_XXXX组_《文档名称》_作者名字.md`
- `XXXX组` 填部门名称，例如：`UGC研发组`
- `文档名称` 填本次需求或方案名称，使用书名号包裹
- `作者名字` 填策划作者姓名
- 生成文档时，应优先按此规则命名，不要临时起松散文件名
- 例如：`U5_UGC研发组_《AI配音升级与语音转换》_初小宇.md`

## 规则章节的常用写法

当需求以规则为主时，优先按下面的顺序组织：

1. 先定义能力范围或权限范围。
2. 再定义开关、状态切换或入口规则。
3. 再定义用户触发动作时的判断链。
4. 再定义失败结果和提示文案。
5. 再定义本次变更会影响什么、不会影响什么。
6. 如果功能依赖实时状态，再补充校验时机。

对于动作时的判断链，要遵守这些写法：

- 通过和失败分支放在一处写清，不要拆散。
- 每个分支都写清“条件”和“结果”。
- 失败分支要带上用户可见提示文案。
- 如果判断通过，只写“进入现有流程”，不要重复赘述下游已知流程。

## Toast与文案规则

- 提示文案保持简短、直接、像产品内真实会出现的文案。
- 只解释用户眼前失败的直接原因，不要解释整套系统。
- 同一个概念在规则、界面文案、示例里要统一叫法。
- 如果有多个失败原因，只有在用户确实需要区分时，才拆成多条不同提示。

## 范围控制

- 优先只写本次变更真正需要的章节，不要机械补齐整套空章节。
- 如果文档结构还没定，就先停下来搭框架，不要直接写正文。
- 如果判断需求会改界面或改场景，而当前没有足够的界面/场景输入信息，就停止写作并向策划索要 Figma、截图、视频或录屏，不要擅自脑补交互。
- 如果用户还在探索期，就先写规则草稿，不要擅自编造交互细节。
- 如果用户问到实现层内容，也优先停留在产品规则和预期层，除非用户明确要技术设计。
- 对后端或服务逻辑，优先描述产品时机和预期结果，不要过度规定内部实现。
- `USLOG/数据埋点` 一般放在最后统一补，不要在前面章节里零散穿插。

## 收尾检查

- 正式文档收尾时，要检查是否需要补 `USLOG/数据埋点` 章节。
- 如果本次功能存在关键行为节点、入口点击、请求成功、保存成功、提审等动作，通常都应进入 `USLOG/数据埋点` 章节。
- 该章节一般作为文档最后一章处理，避免前文规则章节被数据内容打断。

## 交互与原型要求

- 只要需求大概率涉及 UI 界面修改，就必须判断本次会新增哪些页面、弹窗、入口、按钮状态和流程跳转。
- 只要需求大概率涉及游戏内场景改动，就必须判断本次会新增哪些场景表现、操作入口、场景内反馈和状态变化。
- 框架阶段一旦确认这类增删改存在，就应把“是否可以开始画交互原型”作为下一步核心判断。
- 如果现有素材不足以支撑交互章节框架或原型范围判断，就不要继续往下写文档，应先向策划索要界面或场景信息。
- 可接受的信息包括：Figma 原型、静态截图、流程图、录屏、视频、现网界面截图、竞品参考图。
- 只有在交互原型已经足以支撑规则落文档时，才建议进入正式 PRD 正文撰写。

## 参考文档

为了贴近用户现有风格，优先参考以下类型文档：

- 带有 `首页`、`基础规则`、`界面交互流程` 等 sheet 的 Excel 需求文档
- 按日期增量维护的需求文档
- 相邻 UGC 功能的既有需求文档，保证术语和写法一致

在正式起草前，如需快速自检，可阅读 [references/style-guide.md](references/style-guide.md)。
