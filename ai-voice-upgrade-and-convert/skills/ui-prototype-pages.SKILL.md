# UI Prototype Pages

由你自己完成原型实现。除非用户明确要求，否则不要把实现委托给其他模型或外部生成器。

## 工作流

1. 收集最小可用输入。
2. 先做本地可交互原型。
3. 进行视觉和交互验证。
4. 根据截图或反馈返修。
5. 发布到 GitHub Pages。
6. 返回公网 URL 和后续更新路径。

当细节不完整时，优先做合理假设。只有当缺失信息会明显影响布局、导航或发布目标时，才停下来追问。

## 输入信息

接受以下任意组合：

- 参考截图
- 手绘线框图
- Figma 截图或节点链接
- 文字需求
- 目标用户信息，例如 UI、QA、产品或其他评审对象

开始实现前，先识别：

- 关键页面或关键状态
- 必须支持的交互
- 这是一个快速静态稿、多状态原型，还是更接近应用流的网页
- 用户最终是只要本地文件，还是要 GitHub Pages 公网链接

如果用户只给了图片和粗略需求，就根据视觉内容补全默认行为，并在实现后说明自己的假设。

## 实现默认规则

选择能支撑需求的最轻实现方案。

- 如果原型以展示为主，只需要少量点击交互，优先用单文件 HTML。
- 如果原型需要多状态、弹窗、Tab、筛选、列表详情流或后续持续返修，优先用 Vite 加 React。
- 除非用户明确要求真实后端联动，否则使用本地 mock 数据。

尽量保留参考图的视觉意图。如果截图已经体现出明确风格，不要默认做成通用后台或模板化界面。

## 本地构建标准

从零开始时：

1. 为原型创建独立工作目录。
2. 先把 UI 在本地做出来。
3. 确保页面能打开且可交互，再讨论部署。

对 React 原型，默认优先使用：

- vite
- react
- react-dom
- 只添加最少量、确实需要的依赖

避免引入不必要的路由、状态框架或额外构建复杂度。

## 验证标准

在交付前必须自己验证。

至少完成：

- 启动本地页面
- 用浏览器打开
- 检查主交互是否可用
- 对照用户给的截图或需求检查关键视觉

当环境允许时，优先使用 Playwright 来做：

- 关键点击流的烟测
- 对比截图
- 验证 GitHub Pages 发布后的公网地址是否正常打开

对照截图时重点检查：

- 间距和面板比例
- 按钮尺寸和锚点位置
- 预览区表现
- 文本截断和溢出
- emoji、图标、字体 fallback 是否把布局带坏

如果某个视觉问题具有环境相关性，优先用稳定渲染策略解决，而不是用脆弱的内容层补丁。比如把 emoji 或图标放进独立字体栈，而不是依赖页面正文默认字体。

## 返修循环

当用户给出反馈截图时：

1. 明确指出具体不一致点。
2. 找出布局或渲染根因。
3. 直接修代码。
4. 重新构建或刷新。
5. 再次用浏览器验证。

不要只解释问题而不落地修复。必须在原型里改掉并验证，然后用一两句话总结根因。

## 发布标准

当用户要公网链接时，如果满足以下条件，优先使用 GitHub Pages：

- 原型本质上是静态前端
- 浏览器预览足够满足需求
- 用户需要稳定、可分享的 URL

发布流程按下面执行：

1. 先确认项目可以成功构建
2. 配置静态托管所需设置
3. 添加 .gitignore
4. 添加 .github/workflows/ 下的 GitHub Pages 工作流
5. 如有需要，初始化 git 仓库
6. 如果可用，使用 gh 创建 GitHub 仓库
7. 推送 main 分支
8. 用 build_type=workflow 启用 GitHub Pages
9. 等待工作流完成
10. 打开公网 URL 并确认可访问

对于发布到 Pages 的 Vite 项目：

- 如果希望静态资源路径不依赖固定仓库名，优先把 base 设为 ./
- 推送前先跑 npm run build

如果 `gh auth status` 显示已经登录，就自己把仓库创建和发布流程做完；如果没有登录态，就把本地仓库整理好，并告诉用户唯一缺失的那一步。

## 输出标准

完成后返回：

- 本地项目路径
- GitHub Pages 公网 URL（如果已发布）
- GitHub 仓库 URL（如果已创建）
- 一句很短的后续更新方式说明

如果发布成功，在把 URL 给用户之前，必须亲自用浏览器打开验证一次。

## 约束

- 由你自己完成原型实现，不要外包给其他生成器。
- 技术栈保持简单，方便 UI、QA 等非开发角色直接使用。
- 优先保证浏览器直达访问，而不是要求评审同学本地装环境。
- 如果用户要分享链接，优先交付 GitHub Pages 公网地址。
- 不要停在“代码写好了”；要把本地验证和需要时的发布一起做完。
- 如果用户后续问 Figma 导入，要说明 HTML 可以导入成设计内容，但 React 运行时交互不会自动变成 Figma 原生交互。
