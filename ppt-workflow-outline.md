# AI 写需求工作流分享提纲

## 先讲清 OpenSpec 的“上下文”是什么

OpenSpec 的上下文不是隐藏记忆，主要有两层：

1. 方法上下文
- `explore`：把需求盘开，展开规则、状态、边界、待确认项
- `propose`：把探索结果收成结构化骨架

2. 文件上下文
- 如果仓库里已经有 `proposal / design / spec / tasks`
- 后续就能把这些文件继续当成背景材料来复用

当前这个工作区里 `openspec list --json` 是空的，所以这次 OpenSpec 对我来说，主要是规则建模方法，而不是历史上下文引擎。

---

## 这套方法的核心定义

这不是一套“让 AI 帮我写文档”的技巧，
而是一套“把需求从规则、交互到交付分层处理”的工作流。

最重要的锚点：

- 原型不是文档
- GUI 清单不是文档
- 最终 PRD 才是唯一事实源

---

## 整套工作流

```text
一句话需求
   ↓
1. 规则探索
   ↓
2. 结构收敛
   ↓
3. 交互原型
   ↓
4. 正式文档
   ↓
5. GUI交付
   ↓
6. 一致性检查
```

---

## 第 1 步：规则探索

### 目的
- 把一句话需求展开成功能点、状态、边界、待确认项

### 输入示例

```text
我是一个UGC技术策划，我之前做了一个AI配音功能，
允许用户输入文字+选择音色生成音频；
现在我打算升级这个功能到新版的服务让它效果更好，
同时把“语音转换”接入到游戏中，
让用户可以输入语音直接转换成新的音频，在编辑器内使用。
请先盘点一下需求，看要怎么做。
```

### 期望输出
- 功能点拆解
- 状态清单
- 关键链路
- 待确认项
- 边界情况

### 边界
- 不写正式 PRD
- 不出 GUI 清单
- 不急着画原型

### 到什么时候停
- 当我已经能回答：
  - 这个功能有哪些状态
  - 每个状态能做什么
  - 每个按钮点了发生什么
  - 哪些地方还没拍板

---

## 第 2 步：结构收敛

### 目的
- 把规则讨论结果变成稳定结构

### 输入
- 上一步的规则清单、状态、边界、待确认项

### 输出
- proposal / design / tasks
- 或者至少一版不会反复大改的大纲

### 输出示例

```text
1. 功能概述
2. 基础规则
  2.1 服务接口切换
  2.2 新增音色
  2.3 语音转换功能
3. 界面交互
4. USLOG
```

### 边界
- 解决“写什么”
- 不追求最终措辞
- 不直接替代 PRD

### 到什么时候停
- 当我已经有一个稳定的章节树，后面只是填内容，不再反复改框架

---

## 第 3 步：交互原型

### 目的
- 验证状态机和交互，不靠文字脑补

### 输入
- 截图
- 界面描述
- 状态切换说明

### 输出
- 可点击网页原型
- 可被 review 的状态切换
- 可反复截图返修的界面

### 真实案例

```text
点击录音
→ 点击停止
→ 改成按住录音
→ 再改成松手发送、上滑取消
```

### 边界
- 不决定最终规则真相
- 不替代正式文档

### 到什么时候停
- 当关键状态能被看见、被点到、被 review

---

## 第 4 步：正式文档

### 目的
- 把 explore 的规则 + prototype 的交互，合成最终 PRD

### 输入
- 已确认规则
- 已确认原型
- 已确认截图或流程图

### 输出
- 最终需求文档

### 边界
- 文档是最终真相
- 原型和 GUI 清单都不能替代它

### 到什么时候停
- 当开发、QA、GUI 都能从同一份 PRD 找到自己的依据

---

## 第 5 步：GUI 交付

### 目的
- 从 PRD + 原型里，抽出 GUI 真正关心的内容

### 输入
- 最终 PRD
- 网页原型
- 确认后的截图

### 输出
- GUI 交付清单
- 哪些是新增界面
- 哪些是改图
- 哪些需要完整效果图
- 哪些是占位图、资源补充、文字样式说明

### 边界
- 不再重复产品规则
- 不写第二份 PRD

### 到什么时候停
- 当 GUI 拿这份清单可以直接开始，不需要再追问“哪些要出图”

---

## 为什么这套方法有效

1. 先分层，再成文
- 避免一上来把规则、交互、GUI 需求混在一起

2. 每一步都有可见产物
- 不是“和 AI 聊完就没了”，而是每步都有可验证结果

3. 把返工前置
- 很多问题在 explore 和 prototype 阶段就暴露，不会等 PRD 写完再炸

---

## 推广时会卡在哪里

### 方法层问题
- 中间产物太多，容易漂移
- 如果没有“最终事实源”，最后会多版本打架
- 每一步边界不清，就会互相越界

### 环境层问题
- 不是所有人都用 Codex
- 不一定有 OpenSpec CLI
- 不一定会装 skill
- 不一定有 Node / Python / Git / Playwright
- 不一定有 GitHub Pages / SVN 权限
- Windows 路径、中文路径、权限问题很常见

---

## 建议的分层推广方式

### 轻量版
- 一句话需求 -> 规则探索 -> 手工整理 PRD
- 适合没有 Codex、没有完整环境的人

### 标准版
- Explore -> Propose -> PRD -> GUI Brief
- 适合普通策划同学

### 完整版
- Explore -> Propose -> Prototype -> PRD -> GUI Brief -> 一致性检查
- 适合复杂交互需求、高频返修需求

---

## 讲 PPT 时最适合强调的 3 句话

1. 我不是在分享“怎么让 AI 写文档”，而是在分享“怎么把需求生产分层”。
2. OpenSpec 更像规则建模器，不是神秘记忆工具。
3. 最终 PRD 是唯一事实源，原型和 GUI 清单都是围绕它服务的。
